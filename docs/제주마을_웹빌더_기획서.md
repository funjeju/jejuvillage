## 1. 제품 개요

### 1.1 한 줄 정의

> **제주 마을들이 코딩 없이 자기 마을의 관광 홈페이지를 만들고, 실시간 사진 소식·체험상품·예약을 한 곳에서 운영하는 웹빌더 플랫폼.**

### 1.2 배경 & 문제 정의 (확인된 사실 기반)

- 제주에는 다수의 농어촌체험휴양마을이 지정되어 있으나, 개별 마을이 독립 홈페이지·예약 채널을 갖추기 어렵다. (일반적 현황 추론 — 개별 마을 홈페이지 구축률에 대한 정확한 통계는 **확인하지 못함**)
- 조수리 사례처럼 장미마을·빌레용암·480년 팽나무 등 매력 자원이 있어도 SNS에 산발적으로만 노출되고, 통합 브랜드·판매 창구가 없어 체류·결제로 이어지지 않는다.
- 제주도는 2026년 마을만들기 사업에 250억 원 규모를 '콘텐츠 중심'으로 투자한다고 밝혔다. (출처: 제주마을만들기종합지원센터, 2026.01.07) → **디지털 콘텐츠 플랫폼 수요의 정책적 뒷받침 존재**

### 1.3 핵심 가치 제안

| 대상 | 제공 가치 |
|------|----------|
| 마을(운영자) | 코딩 없이 홈페이지 개설, 사진 한 장으로 실시간 소식 발행, 체험상품 등록·예약 관리 |
| 관광객(방문자) | 제주 지도에서 마을 발견 → 살아있는 사진 피드로 '지금의 마을' 확인 → 바로 체험 예약 |
| 행정/지원기관 | 마을별 관광 콘텐츠·예약 데이터의 표준화된 수집·관리 |

### 1.4 제품이 아닌 것 (범위 제외)

- 종합 여행 예약 OTA(항공·렌터카 등)가 아니다.
- 범용 홈페이지 빌더(예: 윅스)가 아니다. **제주 마을 관광**이라는 도메인에 특화한다.
- (MVP 기준) 결제 정산·회계 시스템은 포함하지 않는다. 예약 '신청' 접수까지를 1차 범위로 한다. → **결제 연동 포함 여부는 확정 필요**

## 2. 사용자 & 역할(Role)

### 2.1 사용자 유형

| 역할 | 설명 | 주요 행위 |
|------|------|----------|
| **방문자(Guest)** | 로그인 없이 접근하는 관광객 | 지도 탐색, 피드 열람, 상품 조회, 예약 신청 |
| **마을 운영자(Village Admin)** | 특정 마을의 콘텐츠 관리자(이장·마을기업 담당 등) | 사진/소식 발행, 상품 등록, 예약 확인, 테마·음악 설정 |
| **플랫폼 관리자(Platform Admin)** | 서비스 전체 운영자 | 마을 계정 승인, 템플릿 관리, 전체 통계 |

### 2.2 권한 매트릭스 (요약)

| 기능 | Guest | Village Admin | Platform Admin |
|------|:---:|:---:|:---:|
| 지도·피드·상품 열람 | ✅ | ✅ | ✅ |
| 예약 신청 | ✅ | ✅ | ✅ |
| 자기 마을 소식/상품 CRUD | ❌ | ✅ | ✅ |
| 자기 마을 테마/음악 설정 | ❌ | ✅ | ✅ |
| 예약 내역 확인·상태변경 | ❌ | ✅(자기 마을) | ✅(전체) |
| 마을 계정 생성·승인 | ❌ | ❌ | ✅ |
| 템플릿 추가·수정 | ❌ | ❌ | ✅ |

> **[확정 필요]** 마을 운영자 인증 방식(휴대폰 본인인증 / 행정 발급 코드 / 초대 링크 중 택일). 보안·운영 편의를 고려해 **초대 링크 + 휴대폰 인증** 조합을 제안하나 확정되지 않음.

## 4. 화면별 상세 설계

### S1. 메인 화면 (`/`) — 핵심 화면

**목표**: 방문자가 "제주 어디에 어떤 살아있는 마을이 있는지"를 3초 안에 감지하고, 관심 마을·체험으로 진입하게 한다.

**레이아웃 구성 (위→아래)**

```
┌───────────────────────────────────────────────┐
│  [상단바] 로고 | 마을찾기 | 체험 | 소식 | 로그인   │
├───────────────────────────────────────────────┤
│  [히어로: 인터랙티브 제주 지도]                    │
│   - 제주도 SVG/타일 지도                          │
│   - 마을 위치마다 핀(최근 소식 있으면 핀 애니메이션) │
│   - 핀 hover 시 미리보기 카드(마을명+대표사진+한줄) │
│   - 핀 click → 해당 마을 홈(S3)                   │
│   - 지도 우측/하단: "지금 올라온 소식" 실시간 스트립 │
├───────────────────────────────────────────────┤
│  [실시간 사진 피드]  ← 전체 마을 최신 사진 그리드    │
│   - 카드: 사진 / 마을명 / 시간 / 한줄 캡션          │
│   - 무한 스크롤, 최신순                            │
│   - 카드 click → 마을 홈 해당 소식으로              │
├───────────────────────────────────────────────┤
│  [추천 체험상품 진열]                             │
│   - 가로 스크롤 카드: 썸네일/상품명/마을/가격/소요시간│
│   - "예약" 버튼 → 상품 상세(S4)                    │
├───────────────────────────────────────────────┤
│  [푸터] 서비스 소개 | 마을 입점 문의 | 정책        │
└───────────────────────────────────────────────┘
```

**핵심 인터랙션 명세**

| 요소 | 동작 | 비고 |
|------|------|------|
| 지도 핀 | 최근 N일 내 소식 발행 마을은 핀에 '살아있음' 표시(점멸/링) | N=7 제안 [기획안] |
| 실시간 스트립 | 최신 소식이 좌→우로 흐르거나, 새 소식 도착 시 상단 삽입 | 폴링 or WebSocket |
| 피드 그리드 | 최신순 무한 스크롤. 마을 필터 가능 | 페이지네이션 20개/로드 |
| 추천 진열 | 노출 로직: 예약 많은순 / 신규순 / 수동 큐레이션 | MVP는 최신순+수동 |

**"메인화면 보강 아이디어" (요청 반영 — [기획안])**

1. **계절/시즌 배너**: 조수리 장미철(5~6월)처럼 마을별 '지금이 제철' 콘텐츠를 상단에 자동 노출. (마을이 시즌 태그를 등록하면 해당 기간 가중 노출)
2. **"오늘의 마을" 큐레이션**: 매일 1개 마을을 대표 사진과 함께 히어로에 강조.
3. **지도 ↔ 피드 연동 하이라이트**: 피드 카드에 마우스를 올리면 지도의 해당 마을 핀이 함께 강조되어 '위치 감각'을 준다.
4. **소식→상품 자연 연결**: 소식 사진 하단에 "이 마을 체험 보기" CTA를 붙여 감상→예약 동선을 만든다.
5. **날씨 연동(선택)**: 마을 위치 기반 현재 날씨를 피드 카드에 작게 표시해 방문 판단을 돕는다. → **외부 날씨 API 비용/정책 확인 필요, MVP 제외 권장**

### S3. 마을 홈 (`/v/{slug}`) — 템플릿의 실체

**목표**: 한 마을의 스토리·소식·상품·마스코트를 균형 있게 보여주고 예약으로 연결. **이 화면이 곧 "템플릿"이며, 데이터만 바뀌면 어느 마을이든 렌더링된다.**

**섹션 구성**

```
① 히어로: 마을 대표 이미지 + 마을명 + 한줄 포지셔닝
   - (설정 시) 배경음악 재생 토글 버튼 [♪]
② 마을 스토리: 역사/설화/자원 (탭 또는 스크롤 섹션)
③ 소식 피드: 이 마을의 사진 소식 (최신순)
④ 체험상품: 카드 그리드 → 상세/예약
⑤ 마스코트: 캐릭터 이미지 + 이름 + 소개
⑥ 오시는 길: 지도 + 주소 + 교통/주차 안내
⑦ 푸터: 마을 연락처
```

**템플릿화 핵심 원칙 (개발 지침)**

- 모든 텍스트/이미지/색상/음악은 **DB의 마을 데이터에서 주입**된다. 하드코딩 금지.
- 마을이 특정 섹션 데이터를 비워두면 **해당 섹션은 자동 숨김**(예: 마스코트 미등록 시 ⑤ 섹션 미표시).
- 테마(색상 팔레트·폰트·톤)는 마을별 `theme` 설정으로 스와핑 가능 → 6장·A5 참조.

**배경음악(BGM) 동작 규칙**

- 자동재생은 **하지 않는다**(브라우저 정책·UX 고려). 히어로의 [♪] 버튼을 눌러야 재생.
- 음소거/재생 상태는 세션 유지. 페이지 이탈 시 정지.
- 접근성: 재생 버튼에 명확한 label, 키보드 포커스 지원.

### S4. 체험상품 상세 (`/v/{slug}/product/{id}`)

**구성**: 상품 대표 이미지 갤러리 / 상품명·한줄컨셉 / 가격·소요시간·정원 / 스토리 훅 / 타임라인(구성) / 포함·불포함 / 유의사항 / 리뷰(2차) / **[예약하기] 고정 CTA**

**필수 표기(신뢰 확보)**
- 가격은 세금·수수료 포함 여부 명시.
- 운영 요일/시간, 최소 출발 인원, 취소·환불 규정.
- ⚠️ 규정·가격은 변동 가능 → "최종 확인은 마을에 문의" 안내 문구 노출.

---

### S5. 예약 신청 (`.../book`)

**입력 필드(MVP)**: 날짜 선택(운영일 캘린더) / 인원 / 신청자명 / 연락처 / 요청사항 / 개인정보 수집·이용 동의.

**동작**
1. 제출 → 예약 상태 `REQUESTED`로 저장.
2. 운영자에게 알림(문자/이메일/푸시 중 택1 — **채널 확정 필요**).
3. 신청자에게 접수 확인 안내.
4. 운영자가 A4에서 `CONFIRMED`/`REJECTED`로 상태 변경.

> **[확정 필요]** MVP에서 **실시간 결제**를 넣을지, **신청→수동확정**만 할지. 마을 운영 역량을 고려하면 초기에는 신청제 권장. 결제 도입 시 PG 연동(예: 국내 PG) 별도 설계.

### A2. 소식 발행 (`/admin/feed`) — "사진 한 장으로 소식" 핵심

**목표**: 운영자가 스마트폰으로 **사진 1장 + 한 줄**만으로 즉시 소식을 올린다.

**업로드 폼**
- 사진 첨부(다중 가능, 1~10장) — 카메라 직접 촬영 지원(모바일 `capture` 속성).
- 캡션(한 줄~짧은 글).
- 태그(선택): #장미 #체험 #축제 등.
- 공개 범위: 마을 홈 + 통합 피드(기본) / 마을 홈만.
- [발행] → 즉시 피드 반영.

**관리**: 발행 목록(수정/삭제/숨김), 상단 고정(pin) 기능.

**이미지 처리**: 업로드 시 자동 리사이즈·압축·썸네일 생성(7장 참조). EXIF 위치정보는 **기본 제거**(개인정보), 단 마을 위치 표시는 마을 등록 좌표를 사용.

## 7. 비주얼 디자인 가이드

> **근거**: 이 장의 방향성은 사용자가 제공한 **참고 이미지 2장**에서 도출했습니다.
> - **참고 A (구조·레이아웃)**: 지역 에코투어형 관광 포털로 보이는 웹페이지. 상단 파노라마 일러스트 히어로 + 검색바 / '새소식' 사진 카드 피드 / 체험·장소 썸네일 진열 / "한 눈에 보기" 지도 섹션(핀+라벨 말풍선) / 하단 인스타그램 피드 구조.
> - **참고 B (색·도형·캐릭터)**: 진녹색 배경에 밝은 하늘·구름, 갈색 나무 울타리, 포스트잇 메모, 둥근 말풍선, 마스코트(판다) 캐릭터가 섹션마다 반복 배치된 아기자기한 이벤트 페이지.
>
> ⚠️ **저작권 주의**: 참고 B의 판다 캐릭터는 특정 서비스의 마스코트로 보이나 **정확한 출처는 확인하지 못했습니다(확실하지 않음)**. 캐릭터 자체를 복제하지 말고 **색감·구도·도형 사용 방식(스타일)만** 참고하며, 마스코트는 마을 고유 캐릭터(예: 조수리 '조수이')를 사용합니다. 참고 이미지의 색상 HEX 값은 원본에서 추출한 것이 아니라 **육안 관찰에 기반한 근사치(추정)**이므로, 실제 디자인 확정 시 정확한 값으로 대체해야 합니다.

### 7.1 디자인 컨셉 한 줄

> **"제주 중산간의 초록 들판 위, 아기자기한 마을 그림책" — 자연 친화적 일러스트 톤 + 마스코트가 안내하는 따뜻한 관광 포털.**

### 7.2 컬러 팔레트 (참고 B 기반 — HEX는 추정치, 확정 필요)

| 역할 | 색상(추정) | 용도 |
|------|-----------|------|
| Primary Green (진녹) | 약 `#3E8E41`~`#4CAF50` 계열 | 히어로 배경, 주요 섹션 |
| Sub Green (연녹) | 약 `#8BC34A`~`#AED581` 계열 | 잔디·보조 배경, 버튼 |
| Sky Blue (하늘) | 약 `#B3E5FC`~`#E1F5FE` 계열 | 상단 하늘, 지도 섹션 배경 |
| Earth Brown (울타리·나무) | 약 `#A1725A`~`#C8A27C` 계열 | 나무 울타리, 구분선, 지면 |
| Accent Rose/Red (강조) | 마을 테마색(조수리=장미 레드) | CTA 버튼, 포인트 |
| Cream (메모·카드) | 약 `#FFF9E6`~`#FFFDF5` 계열 | 포스트잇, 카드 바탕 |

> **테마 연동**: 위 팔레트는 **기본(default) 프리셋**이며, 5장 `village_theme`의 `color_*` 필드로 마을별 강조색을 교체한다. 예: 조수리는 accent를 장미 레드로. 즉 **"녹색 자연 톤 + 마을별 포인트 컬러"** 조합.

### 7.3 타이포그래피

| 역할 | 제안 | 비고 |
|------|------|------|
| 디스플레이(제목) | 둥글고 친근한 고딕/손글씨 느낌 한글 폰트 | 참고 B의 말랑한 제목 톤. 예: 배민 주아체 계열 느낌(구체 폰트는 라이선스 확인 필요) |
| 본문 | 가독성 높은 산세리프 한글 폰트 | 예: 프리텐다드 계열(오픈 라이선스 확인) |
| 숫자·데이터 | 본문과 동일 또는 단순 산세리프 | 가격·시간 표기 |

> ⚠️ 구체 폰트명은 **라이선스(웹폰트 사용 허용 여부)**를 반드시 확인 후 확정. 위는 방향 제안일 뿐 확정 아님.

### 7.4 도형·장식 요소 (참고 B 스타일)

참고 B에서 반복되는 아기자기한 도형 언어를 컴포넌트화한다.

| 요소 | 설명 | 사용처 |
|------|------|--------|
| **나무 울타리 구분선** | 갈색 말뚝 울타리 그래픽 | 섹션과 섹션 사이 divider |
| **포스트잇 메모** | 살짝 기울어진 종이 카드(그림자) | 공지·팁·소식 캡션 |
| **둥근 말풍선** | 큰 라운드 말풍선 | 마스코트 대사, 지도 핀 라벨 |
| **구름·잔디·꽃 장식** | 상단 하늘 구름, 하단 잔디/들꽃 | 섹션 배경 상하단 마감 |
| **라운드 코너** | 카드·버튼 큰 둥근 모서리 | 전반적 부드러움 |

> 장식은 **과하지 않게** 사용한다. 참고 B는 이벤트성 페이지라 장식이 많으나, 관광 포털은 정보 가독성이 우선이므로 **핵심 섹션 경계에만 장식**을 배치하고 콘텐츠 영역은 깔끔하게 유지한다.

### 7.5 마스코트 배치 규칙 (참고 B 차용)

- 참고 B처럼 마스코트가 **섹션마다 등장해 안내자 역할**을 한다. 단 캐릭터는 마을 고유(조수이 등).
- 배치 예: 히어로에서 인사 → 소식 섹션에서 "새 소식이 왔어요" → 지도 섹션에서 핀 옆에서 손짓 → 예약 섹션에서 "같이 가요".
- **과다 노출 주의**: 모든 섹션에 크게 넣지 말고, 2~3개 지점에 포인트로.
- 마스코트 이미지는 `village_theme.mascot_media_id`로 주입 → 마을마다 다른 캐릭터로 자동 교체.

### 7.6 메인 화면(S1) 레이아웃 — 참고 A 구조 반영

참고 A의 세로 흐름을 S1에 매핑한다. (4장 S1과 연결)

```
┌─────────────────────────────────────────────┐
│ [상단바] 로고 · 마을찾기 · 체험 · 소식 · 로그인   │
├─────────────────────────────────────────────┤
│ [히어로] 제주 중산간 파노라마 일러스트           │  ← 참고 A 상단 일러스트
│   + 중앙 검색바("어느 마을이 궁금하세요?")        │
│   + 마스코트가 한쪽에서 인사                     │
├─────────────────────────────────────────────┤
│ [새소식] 실시간 사진 카드 피드 (가로/그리드)      │  ← 참고 A '새소식'
│   포스트잇 스타일 캡션 · 마을명 · 시간            │
├─────────────────────────────────────────────┤
│ [체험 진열] 썸네일 카드 가로 스크롤 → 예약        │  ← 참고 A 썸네일 진열
├─────────────────────────────────────────────┤
│ [한 눈에 보기] 제주 지도 + 마을 핀 + 말풍선 라벨   │  ← 참고 A 지도 섹션
│   핀 클릭 → 마을 홈 / 피드 hover 시 핀 강조 연동   │
├─────────────────────────────────────────────┤
│ [SNS 피드] 인스타그램 등 최신 이미지 그리드       │  ← 참고 A 하단 인스타
├─────────────────────────────────────────────┤
│ [푸터] 나무 울타리 장식 + 서비스 정보·입점 문의    │
└─────────────────────────────────────────────┘
```

> **참고 A와의 차이점(의도적 개선)**: 참고 A는 지도가 하단에 있으나, 앞서 사용자가 "메인에서 지도가 나오고"를 강조했으므로 **지도를 히어로 직후 또는 히어로에 통합**하는 안도 검토 가능하다. 두 배치(① 히어로=일러스트+검색 → 소식 → 지도 / ② 히어로 자체를 인터랙티브 지도로)의 최종 선택은 **A/B 검토 후 확정** 권장. → **[확정 필요]**

### 7.7 반응형·접근성 (디자인 필수 기준)

- **모바일 우선**: 참고 A/B 모두 데스크톱 기준이나, 실사용(운영자 현장 사진 발행, 관광객 이동 중 열람)은 모바일 중심. 카드·지도·검색바가 모바일에서 세로로 자연스럽게 쌓이도록 설계.
- **장식 요소는 모바일에서 축소/생략** 가능하게(성능·가독성).
- 명도 대비: 녹색 배경 위 텍스트는 충분한 대비 확보(흰 카드 위 진한 텍스트 원칙).
- 모션(구름 흐름 등)은 `prefers-reduced-motion` 존중.

## 8. 미디어 처리 (사진/음악)

### 8.1 이미지

- **허용 형식**: JPG, PNG, WebP. (HEIC는 서버에서 변환 권장 — iOS 대응)
- **업로드 제한(제안)**: 장당 최대 10MB, 소식당 최대 10장. → **[확정 필요]**
- **처리 파이프라인**: 브라우저 → **Firebase Storage 직접 업로드** → **Cloud Function 트리거**가 리사이즈(예: 최대 1600px 장변)·WebP 변환·썸네일(예: 400px) 생성 → URL을 Firestore에 기록 → Vercel 엣지/`Next/Image`로 전송 최적화.
- **EXIF**: 위치정보(GPS) 기본 제거. 회전정보(Orientation) 반영 후 정규화.
- **저장소**: **Firebase Storage**. 공개 읽기는 published 마을 자산만 허용(Storage 보안규칙).

### 8.2 음악(BGM)

- **허용 형식**: MP3, (선택)AAC. 용량 제한 제안: 최대 8MB / 최대 3분. → **[확정 필요]**
- **재생 규칙**: 자동재생 금지, 사용자 토글 재생, 루프 옵션(마을 설정).
- **저작권 주의(중요)**: 업로드되는 음악의 **저작권 책임은 업로더(마을)에게 있음**을 약관·업로드 화면에 명시하고 동의를 받는다. 플랫폼은 저작권 침해 신고 처리 절차(신고→검토→비공개)를 갖춘다. → 상용 음원 무단 업로드 방지 안내 필수.

### 8.3 실시간 피드 갱신 방식

- **Firestore 실시간 리스너(onSnapshot)** 로 새 소식 준실시간 반영이 기본 가능. 단 리스너 남용은 읽기 비용을 키우므로 노출 범위를 제한한다.
- **MVP 권장**: 첫 로드는 쿼리, 이후 갱신은 화면에 보이는 목록에만 제한적 리스너 또는 폴링(30~60초).
- 통합 피드의 전역 조회 비용은 9.1.2절의 Collection Group vs 미러링 결정과 연계.

## 9. 기술 스택 제안 & 비기능 요구사항

### 9.1 기술 스택 — Vercel + Firebase 구조 (확정 방향)

> 사용자 요청에 따라 **Vercel(프론트엔드/호스팅) + Firebase(백엔드 서비스)** 조합으로 확정합니다. 아래는 각 서비스의 역할 분담입니다.

| 영역 | 사용 서비스 | 역할 |
|------|------------|------|
| 프론트엔드 프레임워크 | **Next.js** (App Router) | 마을 홈 SSR/ISR로 SEO 확보, 어드민은 CSR |
| 호스팅·배포 | **Vercel** | Next.js 앱 배포, 프리뷰 배포, 엣지 CDN, 도메인 관리 |
| 인증 | **Firebase Authentication** | 운영자·플랫폼관리자 로그인. 휴대폰(SMS) 인증, 이메일/링크 로그인 지원 |
| 데이터베이스 | **Cloud Firestore** | 마을·소식·상품·예약 등 문서형 저장 (5장 스키마를 컬렉션으로 매핑) |
| 파일 저장 | **Firebase Storage** | 사진·음악 원본/썸네일 저장 |
| 서버 로직 | **Cloud Functions for Firebase** | 이미지 리사이즈, 예약 알림 발송, 권한 검증 등 트리거·API |
| 이미지 최적화 | **Next/Image + Vercel** 또는 Storage 리사이즈 | 두 방식 병행 가능(아래 8장·7절 참조) |
| 푸시/알림 | **FCM**(선택) + 외부 SMS | 운영자 예약 알림. SMS는 외부 발송사 필요(아래 주석) |

**Vercel과 Firebase Hosting의 경계 (혼동 주의)**
- **Firebase Hosting은 사용하지 않고**, 앱 호스팅은 **Vercel로 일원화**한다. (Next.js는 Vercel에서 가장 매끄럽게 동작)
- Firebase는 **백엔드 서비스(Auth·Firestore·Storage·Functions)만** 사용한다.
- → 이렇게 나누면 배포는 Vercel, 데이터·인증·파일은 Firebase로 책임이 명확해진다.

**아직 확정되지 않은 외부 연동 (확실하지 않음)**
- **SMS 본인인증/예약 알림 발송사**: Firebase Auth의 전화 인증은 로그인용이며, 예약 접수 알림 문자를 보내려면 별도 SMS 발송 서비스(국내 문자 API 등)가 필요하다. 구체 벤더·비용은 **확인되지 않음** → 확정 필요.
- **지도 SDK**: Firebase/Vercel과 무관하게 별도 선정 필요(제주 커버리지·라이선스·비용 비교). 특정 벤더 단정하지 않음.
- **결제(PG)**: MVP 신청제 기준이면 불필요. 도입 시 국내 PG 별도 연동.

### 9.1.1 시스템 아키텍처 (개념도)

```
[방문자/운영자 브라우저]
        │
        ▼
┌───────────────────────────┐
│  Vercel (Next.js)          │
│  - 공개: 마을 홈 SSR/ISR     │
│  - 어드민: CSR(로그인 후)     │
│  - Route Handlers(API)      │  ── 서버측에서 Firebase Admin SDK 사용
└───────────┬───────────────┘
            │ (Firebase SDK / Admin SDK)
   ┌────────┼─────────┬───────────┬─────────────┐
   ▼        ▼         ▼           ▼             ▼
[Auth]  [Firestore] [Storage]  [Functions]   [FCM]
로그인    문서 DB     사진/음악   이미지리사이즈  푸시알림
권한      마을데이터            예약알림트리거
```

**요청 흐름 예시 — "운영자가 사진 소식 발행"**
1. 운영자가 어드민에서 사진 선택 → 브라우저가 **Firebase Storage에 직접 업로드**(서명 URL 또는 SDK).
2. 업로드 완료 → **Cloud Function 트리거**가 리사이즈·WebP·썸네일 생성 후 URL을 Firestore `feed_post`에 기록.
3. 공개 피드는 Firestore 실시간 리스너(onSnapshot) 또는 폴링으로 갱신.

**요청 흐름 예시 — "방문자가 예약 신청"**
1. 예약 폼 제출 → Vercel Route Handler(또는 Function)가 검증 후 Firestore `booking`에 `REQUESTED`로 저장.
2. `booking` 생성 트리거 Function이 해당 마을 운영자에게 알림(FCM/SMS).
3. 운영자가 어드민에서 상태를 `CONFIRMED`/`REJECTED`로 변경.

### 9.1.2 Firestore 데이터 구조 (5장 스키마의 NoSQL 매핑)

> 5장은 관계형(테이블) 기준으로 작성되었으므로, Firestore(문서형)로 옮길 때의 컬렉션 구조를 아래처럼 제안한다. **[기획안]**

```
villages/{villageId}
  - name, slug, region, lat, lng, one_liner, status, ...
  villages/{villageId}/stories/{storyId}
  villages/{villageId}/feed_posts/{postId}
      - caption, tags[], visibility, is_pinned, published_at, mediaRefs[]
  villages/{villageId}/products/{productId}
      - title, concept, price, duration_min, timeline[], status, ...
  villages/{villageId}/bookings/{bookingId}
      - productId, date, headcount, applicant_name, phone, status, ...
  villages/{villageId}/theme (단일 문서)
      - preset_key, color_*, mascot_name, bgm_url, ...

media_assets/{assetId}   (또는 마을 하위)
  - villageId, type(image/audio), url, thumb_url, meta

users/{uid}
  - name, phone, email, role(guest/village_admin/platform_admin)
  - managedVillages[]   (운영 가능한 마을 id 목록)
```

- **통합 피드**(메인 S1·통합 /feed)는 마을 하위 컬렉션을 가로질러 조회해야 하므로, Firestore의 **Collection Group Query**로 `feed_posts`를 전역 조회하거나, 별도 `global_feed` 컬렉션에 미러링하는 방식을 검토한다. → **성능·비용 트레이드오프 확정 필요**.
- Firestore는 조인이 없으므로, 목록 화면에서 필요한 필드는 **의도적으로 중복 저장(비정규화)**한다(예: `booking`에 villageName 함께 저장).

### 9.1.3 어드민 모드 상세 (요청 반영: "생성된 마을 홈페이지 관리")

> 기존 4장 A1~A6(마을 운영자)·P1~P3(플랫폼 관리자) 화면을 Firebase 권한 체계로 구체화한다.

**두 단계 어드민**

| 레벨 | 접근 경로 | 관리 범위 |
|------|----------|----------|
| **마을 어드민**(village_admin) | `/admin` | 자기 마을의 소식·상품·예약·테마·기본정보 |
| **플랫폼 어드민**(platform_admin) | `/platform` | **생성된 모든 마을 홈페이지 관리**: 마을 계정 생성·승인·정지, 마을 목록 조회, 콘텐츠 모더레이션(신고 처리), 템플릿 관리, 전체 통계 |

**플랫폼 어드민 — "마을 홈페이지 관리" 핵심 기능(요청의 중심)**

| 기능 | 설명 |
|------|------|
| 마을 목록·검색 | 생성된 모든 마을을 상태(draft/published/정지)별로 조회 |
| 마을 생성·초대 | 새 마을 개설 → 운영자에게 초대 링크 발송 |
| 게시/비공개 토글 | 특정 마을 홈페이지를 공개/비공개 전환 |
| 콘텐츠 모더레이션 | 신고된 소식·음악(저작권) 검토 후 숨김/삭제 |
| 계정 정지·권한 회수 | 문제 운영자 계정 정지, 관리 권한 회수 |
| 템플릿·프리셋 관리 | 7장 테마 프리셋 추가·수정, 마을에 배포 |
| 전체 통계 | 마을별 방문·소식·예약 수 대시보드 |

**권한 제어 방식 (Firebase)**
- 역할은 **Firebase Auth Custom Claims**(`role`, `managedVillages`)에 저장 → 클라이언트·서버·보안규칙에서 공통 참조.
- 권한 부여/회수는 **플랫폼 어드민만** 가능(Cloud Function 통해 Custom Claims 설정).
- 아래 Firestore 보안 규칙으로 최종 방어.

### 9.1.4 Firestore 보안 규칙 (핵심 원칙 — [기획안])

> 실제 규칙은 개발 시 정밀화. 아래는 **접근 통제 의도**를 규칙 의사코드로 표현한 것.

```
match /villages/{vid} {
  allow read: if resource.data.status == 'published' || isAdminOf(vid) || isPlatformAdmin();
  allow write: if isAdminOf(vid) || isPlatformAdmin();

  match /feed_posts/{pid} {
    allow read: if published or admin;
    allow create, update, delete: if isAdminOf(vid) || isPlatformAdmin();
  }
  match /bookings/{bid} {
    allow create: if true;              // 방문자 예약 신청 허용
    allow read, update: if isAdminOf(vid) || isPlatformAdmin();  // 열람·상태변경은 운영자만
  }
}
function isAdminOf(vid): request.auth.token.role == 'village_admin'
                         && vid in request.auth.token.managedVillages
function isPlatformAdmin(): request.auth.token.role == 'platform_admin'
```

- **개인정보 보호**: `bookings`의 신청자 연락처는 운영자·플랫폼 어드민만 읽을 수 있어야 한다(위 규칙). 방문자는 create만 가능, read 불가.
- **Storage 규칙**: 업로드는 인증된 운영자만, 공개 읽기는 published 마을 자산만 허용하도록 별도 Storage 보안규칙 작성.

### Phase 2 — 플랫폼화 (P2)

### 지원사업 연계 (확인된 사실 기반)
